# üîπ Migration to Azure SQL ‚Äì IaaS vs PaaS

## **1. Migration to IaaS (SQL Server on Azure VMs)**

üëâ Basically a **lift-and-shift** approach. You‚Äôre just moving your existing SQL Server into an Azure VM.

| Aspect             | IaaS Migration                                                                                                                                                                                                                                                                       |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Strategy**       | ‚úÖ *Rehost* (Lift-and-shift) <br> ‚úÖ *Replatform* (minimal changes, e.g. disks/storage config)                                                                                                                                                                                         |
| **Typical Tools**  | - **Azure Migrate** (for VM discovery & lift-and-shift) <br> - **Backup/Restore** (native SQL backup to Azure Blob, then restore on VM) <br> - **Log Shipping** (for cutover with minimal downtime) <br> - **Database Mirroring / Always On AGs** (sync databases before switchover) |
| **Pros**           | - Fastest migration path <br> - Full SQL feature support (Agent jobs, SSIS, cross-DB queries, CLR, etc.) <br> - Minimal app changes                                                                                                                                                  |
| **Cons**           | - You still manage OS, patching, HA/DR setup <br> - Not fully ‚Äúcloud native‚Äù <br> - Higher TCO if not optimized                                                                                                                                                                      |
| **When to Choose** | - If you need **full SQL feature parity** (e.g., SSRS, SSAS, cross-instance features) <br> - Legacy apps tightly coupled with SQL Server <br> - Quick lift needed with minimal code change                                                                                           |

## **2. Migration to PaaS (Azure SQL Database / Managed Instance)**

üëâ Here you‚Äôre **replatforming/refactoring** because PaaS doesn‚Äôt support every SQL feature.

| Aspect             | PaaS Migration                                                                                                                                                                                                                                                                                                                                                                                               |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Strategy**       | ‚úÖ *Replatform* (DB schema + data move to PaaS) <br> ‚úÖ *Refactor* (app code changes if features unsupported)                                                                                                                                                                                                                                                                                                  |
| **Typical Tools**  | - **Data Migration Assistant (DMA)** ‚Üí Assess compatibility, find unsupported features <br> - **Azure Database Migration Service (DMS)** ‚Üí Online/offline migrations <br> - **BACPAC Export/Import** ‚Üí Schema + data export (good for smaller DBs) <br> - **Transactional Replication** ‚Üí Push data into Azure SQL DB <br> - **Log Replay Service (MI only)** ‚Üí Continuous log shipping for large migrations |
| **Pros**           | - Fully managed (HA, backups, patching) <br> - Lower TCO, elastic scaling <br> - Cloud-native features (geo-replication, serverless, hyperscale)                                                                                                                                                                                                                                                             |
| **Cons**           | - Limited SQL features (no SQL Agent in SQL DB, limited cross-db queries, some CLR/SSRS not supported) <br> - App refactoring may be required <br> - Migration effort higher                                                                                                                                                                                                                                 |
| **When to Choose** | - If you want **minimal ops overhead** <br> - Apps can work within **PaaS limitations** <br> - Need cloud-native scaling (Hyperscale, serverless) <br> - Targeting SaaS, multi-tenant, or modern workloads                                                                                                                                                                                                   |

### üîë Quick ‚ÄúWhen to Use‚Äù Summary

* **VPN Gateway** ‚Üí Use for secure hybrid connectivity (encrypted tunnel over internet).

* **Service Endpoint** ‚Üí Use when you want to restrict access by VNet/subnet but can still tolerate public endpoint.
* **Private Link** ‚Üí Use when you must kill public endpoints and route through private IP only.

# üìò Query Store Cheat Sheet (Azure SQL Database)

---

## üîπ 1. What Query Store Does

* Captures query **execution plans** and **runtime stats** over time.
* Helps analyze **performance regressions**, **plan changes**, and **slow queries**.
* Persists across restarts (unlike DMVs).

---

## üîπ 2. Query Store States

* **Read-Write** ‚Üí Normal, captures data.
* **Read-Only** ‚Üí Captures stop, you can still query history.
* **Off** ‚Üí Disabled.

### Why Query Store goes **Read-Only**

1. Database runs out of space for Query Store.
2. Data flushes from memory to disk take too long.
3. IO or resource pressure.

üëâ **Fix:** Adjust flush interval, increase database size, or clean old data.

## üîπ 3. Key Configuration Settings

* **Operation Mode**:

  * Off / Read-Only / Read-Write
* **Data Flush Interval** (default: 15 min):

  * How often in-memory stats get written to disk.
  * üèÜ *Reduce this to prevent read-only transitions.*
* **Statistics Collection Interval** (default: 1 hr):

  * Controls aggregation of runtime stats, not related to read-only.
* **Max Size (MB)**:

  * Cap for Query Store data. If reached ‚Üí goes Read-Only.
* **Stale Query Threshold (days)**:

  * Auto-cleanup of old query data.

## üîπ 4. Monitoring Query Store

* **DMVs**:

  * `sys.database_query_store_options` ‚Üí Current settings.
  * `sys.query_store_runtime_stats` ‚Üí Runtime stats.
  * `sys.query_store_plan` ‚Üí Plans captured.
  * `sys.query_store_query_text` ‚Üí Query text.

* **Query Performance Insight (QPI)** in Azure Portal:

  * High CPU, IO-heavy queries, regressed queries.

* **Extended Events / DMVs** ‚Üí Deep diagnostics.

## üîπ 5. Query Store Use Cases in Exams

1. **Identify if tempdb is a bottleneck** ‚Üí Use **DMVs**.
2. **Identify top resource-consuming queries** ‚Üí Use **Query Performance Insight**.
3. **Prevent Query Store going Read-Only** ‚Üí Decrease **Data Flush Interval**.
4. **Capture query regressions** ‚Üí Use **Query Store forced plan**.
5. **Turn on Query Store** ‚Üí `ALTER DATABASE SET QUERY_STORE = ON`.

## üîπ 6. Best Practices

* Keep **flush interval low** (5‚Äì10 min) for high workload DBs.
* Set **Max Size (MB)** high enough for workload.
* Use **cleanup policies** to purge old/stale queries.
* Use **forced plans** carefully (good for regressions, risky for evolving workloads).
